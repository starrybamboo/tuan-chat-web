# 团剧共创宣传页设计讨论稿

> 状态：讨论稿
> 目标：先确定宣传页的叙事、信息架构、视觉方向和动效策略，再进入正式实现。

参考: https://anime.ai/

## 1. 页面目标

介绍团剧共创的基本操作逻辑和设计思路？



- 团剧共创是什么。
- 它解决了什么创作流程问题。
- 它能把一次聊天现场转化成什么成果。
- 用户下一步应该进入哪里。


## 2. 核心主张

候选主标题：

> 从一段发言，踏进共创世界。

候选副标题：

> 把角色、聊天、素材、分镜和演出放进同一个创作现场，让一次即兴共创沉淀成可回看、可扩写、可继续演出的团剧。

需要讨论：

- “团剧共创”这个词是否需要在首屏解释。
- 首屏更强调“从聊天到演出”，还是更强调“多人协作创作空间”。
- 目标用户是跑团玩家、剧本作者、素材创作者，还是泛二创社区用户。

## 3. 用户认知路径

建议页面按以下认知路径推进：

1. 用户先看到一个“故事正在生成”的沉浸式场景。
2. 用户理解当前创作工具的问题：聊天、设定、素材、预览分散。
3. 用户看到团剧共创如何把这些内容组织到同一条叙事线上。
4. 用户看到产物：角色、分镜、演出、归档、公开素材。
5. 用户被引导进入房间或探索素材。

关键不是展示所有功能，而是展示“创作从混乱到成篇”的变化。

## 4. 推荐页面结构

| 区块 | 目标 | 主要内容 | 动效建议 |
| --- | --- | --- | --- |
| Hero | 建立第一印象 | 一段发言生成角色卡、分镜碎片、舞台光 | sticky 首屏，Canvas 序列帧 |
| Problem | 说明痛点 | 聊天、素材、设定、演出分散 | 文案逐条浮现 |
| Flow | 展示创作流程 | 写下第一句、接住分支、同步画面、归档成篇 | 横向滚动卡片 |
| Product Proof | 证明这是工具 | 房间、角色、素材库、WebGAL 预览、归档 | 分层 UI 面板组合 |
| Community | 展示长期价值 | 公开素材、Fork、模组沉淀、再创作 | 关系网络或资产流动 |
| CTA | 引导行动 | 进入房间、探索素材、创建空间 | 简洁强按钮 |

## 5. 视觉方向

推荐方向：

> 暗色舞台 + 编辑器 HUD + 动画分镜。

这个方向适合团剧共创，因为它同时包含：

- “剧场感”：角色入场、舞台、灯光、演出。
- “工具感”：时间线、卡片、面板、状态同步。
- “创作感”：分镜、素材、旁白、对话节点。

视觉关键词：

- 夜间剧场
- 霓虹边缘光
- 半透明编辑器面板
- 分镜纸 / Storyboard
- 角色卡牌
- 舞台光锥
- 轻微颗粒纹理
- 叙事时间线

不建议：

- 普通 SaaS 白底功能卡片。
- 过度通用的紫色渐变科技风。
- 把页面做成单纯的动画炫技。

## 6. 动效策略

当前技术验证结论：

- Motion 可以承担正式宣传页动效。
- 可以使用 `useScroll` 驱动页面进度。
- 可以使用 `position: sticky` 实现类 pin 效果。
- 可以使用 `useTransform` 做透明度、位移、缩放和横向滚动。
- Canvas 可以根据 scroll progress 渲染程序化序列帧。

正式实现建议：

- 动效服务叙事，不要所有元素都动。
- 首屏动效可以强，后续区块应该更清晰。
- 每个滚动段落只表达一个状态变化。
- 移动端优先保证阅读，复杂横向滚动可以降级为纵向卡片。
- 减少持续重绘区域，Canvas 控制 DPR，避免低端设备掉帧。

## 7. 文案方向

文案应该更像“创作现场的描述”，少用抽象产品词。

候选短句：

- 一句发言，也可以成为下一格画面。
- 角色、素材、分镜和演出，不再散落在不同窗口。
- 把临场灵感沉淀成可以回看、扩写和继续演出的团剧。
- 从聊天现场到动画分镜，创作过程本身就是舞台。
- 让每一次共创，都留下可以继续生长的世界。

需要避免：

- “提升效率”“赋能创作者”“一站式平台”这类通用 SaaS 话术。
- 太多概念名词堆叠。
- 过早展示复杂功能细节。

## 8. 首屏方案草案

首屏可以这样组织：

- 左侧：核心标题、副标题、两个 CTA。
- 右侧 / 背景：角色卡、聊天气泡、分镜框、舞台光逐渐聚合。
- 滚动时：首屏文案淡出，角色卡和分镜碎片进入更完整的舞台结构。

首屏信息优先级：

1. “从一段发言，踏进共创世界。”
2. “聊天、角色、素材、分镜、演出在同一条叙事线上汇合。”
3. 主 CTA：进入团剧共创。
4. 次 CTA：探索素材。

## 9. Flow 区块草案

建议保留四步：

1. 写下第一句
   从角色视角发言，保留现场语气与行动意图。

2. 接住分支
   把选择、检定与临场补充留在同一条叙事线上。

3. 同步画面
   素材和演出参数跟随消息流进入预览舞台。

4. 归档成篇
   完整片段沉淀为可回看、可复用、可继续创作的内容。

需要讨论：

- 是否要加入“邀请成员”或“创建空间”步骤。
- 是否要把 WebGAL 预览明确写进流程。
- 是否要把“公开素材 / Fork”放在流程中，还是放在后面的社区价值区块。

## 10. Product Proof 区块草案

这个区块要从抽象故事回到真实产品能力。

建议展示的能力：

- 房间：多人实时共创。
- 角色：设定、头像、立绘、关系。
- 素材：公开素材库和空间素材。
- 分镜：消息流转换为结构化片段。
- 演出：WebGAL 预览或舞台呈现。
- 归档：从一次创作沉淀为可继续使用的作品。

表现形式：

- 不一定要使用真实截图。
- 可以用“产品 UI 抽象化面板”表达真实能力。
- 如果使用真实截图，需要避免截图信息过密，应该做局部放大和叠层解释。

## 11. CTA 策略

主 CTA 候选：

- 进入团剧共创
- 创建你的第一间房
- 开始下一幕

次 CTA 候选：

- 探索公开素材
- 查看创作空间
- 了解素材库

需要讨论：

- 对新用户来说，“进入房间”是否足够明确。
- 是否需要根据登录状态显示不同 CTA。
- 是否要把素材库作为更低门槛入口。

## 12. 技术实现边界

已验证可以做：

- Motion 滚动驱动。
- 原生滚动 + sticky pin。
- 横向滚动卡片。
- Canvas 序列帧。
- 独立宣传页滚动容器处理。

需要注意：

- 全局 `body` 默认锁滚动，宣传页需要独立打开原生滚动。
- 移动端 sticky 高度和横向卡片要单独处理。
- Canvas 渲染要控制性能，不要在非可见区域持续重绘。
- 宣传页最好保持为独立路由，不被主应用布局和聊天滚动逻辑影响。

## 13. 待定问题

- 首屏是否应该出现真实产品 UI？
- 页面整体偏“剧场感”还是偏“编辑器工具感”？
- 是否需要加入用户案例或创作成果展示？
- 是否要突出 WebGAL 作为演出能力？
- 是否需要展示社区资产流转：公开、Fork、再创作？
- 移动端是否保留复杂滚动叙事，还是改成更直接的阅读流？

## 14. MVP 建议

第一版建议只做五段：

1. Hero：从一段发言进入共创世界。
2. Pain：创作信息分散的问题。
3. Flow：聊天到分镜到演出的流程。
4. Proof：角色、素材、预览、归档的产品能力。
5. CTA：进入团剧共创。

第一版验收标准：

- 用户能在 30 秒内说出团剧共创是做什么的。
- 页面视觉有“团剧 / 舞台 / 分镜”的记忆点。
- 滚动动效不妨碍阅读。
- 移动端能完整阅读并顺利点击 CTA。
