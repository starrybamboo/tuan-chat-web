# 咕噜噜安科文 Replay 导入方案

## 目标

将 `D:\gululu-cache\output\opus-88\parts` 中的楼层 Markdown 转换为团剧共创的消息流，再复用现有 WebGAL 导出能力生成 replay。

第一版试点范围为《烈海王似乎打算在幻想乡挑战强者们的样子》的第 1 到 62 楼。

## 输入结构

源目录包含：

- `parts/*.md`：按楼层范围切分的 Markdown。
- `images/**`：楼层正文引用的图片资源。
- `meta.json`：作品标题、作者、楼层数量等元信息。
- 图片分类报告：仅作为参考，不作为角色身份的唯一依据。

Markdown 楼层通常具有以下结构：

```markdown
## 第7楼

> 时间: 2022-01-22 20:40

神子的急救【1d100:90】
（90以上不需要永琳就能救命了）
![image](../images/刃牙/烈海王/254.png)
烈，你捡了一条命呢
```

## 核心原则

### 楼层不是演出消息

楼层号、时间、作者、来源不进入 WebGAL 演出文本。

这些信息只保存在导入包元数据里，用于校对、回跳、排查错配。

### 按事件块拆分

不采用“一楼一条消息”。一楼内可能包含多张头像、多个角色对白、骰子判定和旁白说明。

导入时应拆成 replay 事件流：

```text
[骰娘]：神子的急救【1d100:90】
[旁白]：（90以上不需要永琳就能救命了）
[神子]：烈，你捡了一条命呢
```

### 显式说话人优先

如果文本中出现 `角色名：内容`，以文本中的角色名为准。

图片路径只作为头像候选，不直接决定角色名。原因是现有图片分类可能有误，例如某些视觉上是神子的头像被归类到了 `刃牙/烈海王` 目录。

### 图片反向学习角色

导入器会统计：

```text
图片 A 多次出现在 “神子：...” 的文本段旁
=> 图片 A 可推断为神子的头像/表情图
```

当后续遇到同一张图但文本没有显式 `神子：` 前缀时，可以推断为神子发言。

推断规则保持保守：数字选项、长段说明、规则说明优先保持旁白，避免误绑角色。

## 消息类型映射

### 对白

显式对白：

```text
神子：在神灵庙的门口出现了……
```

导入为：

```text
kind: dialog
speakerName: 神子
content: 在神灵庙的门口出现了……
imagePath: 当前图片
```

### 旁白

无角色前缀、也不是骰子/BGM 的叙述文本导入为旁白。

数字选项列表如果没有紧邻前置骰子，则保持旁白。

### 骰子

包含 `【1d100:90】`、`【1d10：10】`、`【262+1d100：82=344】` 等格式的行导入为骰娘消息。

导入时不重新投骰，只保留原文和原结果。

骰子后紧跟的数字选项列表应绑定进同一条骰娘消息，显示时不折叠：

```text
[骰娘]：那么烈啊，你要去往何处呢【1d13:9】
1 博丽神社
2 红魔馆（是红海皇！）
...
13 其他的势力
```

### BGM

`BGM：xxx` 导入为独立 BGM 消息。

第一版规则：

- 优先匹配本地 BGM 目录或 manifest。
- 未匹配到音频时保留缺失清单。
- 不从未授权平台自动抓取音频。

WebGAL 导出时映射为 BGM 切换。

## 角色卡与头像

导入时生成一套团剧角色卡：

- 按最终识别出的 `speakerName` 创建角色。
- 每个角色绑定多张头像/表情图。
- 消息绑定当前使用的头像，而不是只绑定默认头像。

示例：

```text
角色：神子
  默认头像：出现次数最多的神子图
  表情图：
    东方/丰聪耳神子/654.png
    刃牙/烈海王/254.png
```

## 导入包产物

第一版脚本会生成两个文件：

```text
D:\gululu-cache\output\opus-88\tuanchat-import\opus-88-floors-1-62.tuanchat-replay-import.json
D:\gululu-cache\output\opus-88\tuanchat-import\opus-88-floors-1-62.chat-import.txt
```

JSON 是完整导入包，包含：

- `messages`：演出消息流。
- `roles`：自动识别出的角色与头像候选。
- `source`：楼层范围、作品标题等元信息。
- `stats`：消息数量、角色数量、推断对白数量。

TXT 是兼容现有“导入对话”窗口的预览格式，用于快速校对文本切分结果。

## 第一版试点结果

第 1 到 62 楼当前生成：

```text
totalMessages: 283
dialog: 129
dice: 53
narration: 101
inferredDialog: 49

```

这些数字说明：

- replay 不能按楼层导入，否则会损失楼内节奏。
- 骰子在早期楼层中已经是核心叙事结构。
- 头像反推有价值，但必须保守。

## 后续接入路径

### 第一阶段：导入包生成与人工校对

使用脚本生成导入包和预览文本。

人工检查：

- 角色名是否错配。
- 图片是否绑定到正确角色。
- 骰子与选项是否合并正确。
- BGM 是否需要补本地音频。

### 第二阶段：接入团剧导入 UI

新增“Replay 导入”入口：

- 选择源目录。
- 选择楼层范围。
- 展示角色/头像映射预览。
- 展示缺失 BGM 清单。
- 确认后创建角色卡、头像和消息流。

### 第三阶段：WebGAL 导出增强

让 WebGAL 导出识别 replay 导入产生的元数据：

- 对白使用对应角色头像/立绘。
- 骰娘消息保留完整骰子文本和选项。
- BGM 消息切换背景音乐。
- 旁白作为普通叙述文本输出。

## 当前限制

- 现有脚本只生成导入包和文本预览，尚未直接调用后端创建角色/上传头像/发送消息。
- BGM 只做语义识别，暂不自动下载音频。
- 头像反推是启发式算法，仍需要校对。
- 含有松散引号的台词格式仍可能需要继续调优。
