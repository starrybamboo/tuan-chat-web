# 角色图片裁剪与媒体文件 PRD

## 背景

角色图片链路需要同时满足两个运行场景：

1. Web 端使用按需派生的压缩图片，控制加载速度和带宽成本。
2. WebGAL 合成、桌面端打包和后续编辑需要稳定的媒体文件 ID，不能依赖长期保存的图片地址字段。

头像裁剪必须建立在立绘裁剪结果之上，不能直接从最初上传图裁头像。

## 目标

1. 角色头像记录只保存 `originFileId`、`spriteFileId`、`avatarFileId` 等媒体文件 ID。
2. 展示、预览、导出所需的图片地址在运行时通过媒体文件 ID 派生。
3. 上传与批量导入流程统一为：上传原图、立绘校正、头像校正。
4. 立绘组的合成配置绑定同一套裁剪上下文，加入立绘组时使用组配置覆盖单个头像自己的上下文。

## 非目标

1. 不保留旧图片地址字段的运行时兼容分支。
2. 不在角色头像实体上新增长期保存的展示地址字段。
3. 不实现完整的立绘局部替换编辑能力，只保留其所需的数据前提。

## 字段语义

| 字段 | 语义 | 主要用途 |
| --- | --- | --- |
| `originFileId` | 用户上传的原始图片媒体文件 ID | 立绘校正输入、源文件留存 |
| `spriteFileId` | 立绘裁剪结果媒体文件 ID | 立绘展示、头像裁剪输入、WebGAL 立绘层 |
| `avatarFileId` | 头像裁剪结果媒体文件 ID | 头像展示、头像层合成、小头像派生 |
| `spriteCropContext` | 原图到立绘的裁剪上下文 | 复现立绘裁剪、批量应用 |
| `avatarCropContext` | 立绘到头像的裁剪上下文 | 复现头像裁剪、立绘组合成 |
| `spriteTransform` | WebGAL 立绘显示变换 | WebGAL 预览和发布 |
| `variantGroup.compositionConfig` | 立绘组头像覆盖槽位配置 | 同组头像/立绘合成 |

## 核心规则

1. `originFileId` 只用于立绘校正输入，不作为头像裁剪源。
2. 头像裁剪源必须是 `spriteFileId` 对应的立绘裁剪结果。
3. 没有 `spriteFileId` 时，头像裁剪应提示先完成立绘裁剪。
4. 加入立绘组时，头像裁剪上下文和 WebGAL transform 使用立绘组配置，不保留单个头像自己的旧配置。
5. 立绘组内的原图尺寸必须一致，否则不能套用同一组合成配置。

## 用户流程

| 步骤 | 用户动作 | 系统输出 |
| --- | --- | --- |
| 1 | 上传一张原始图片 | 创建带 `originFileId` 的头像记录 |
| 2 | 对原图进行立绘校正 | 写入 `spriteFileId` 和 `spriteCropContext` |
| 3 | 进入头像校正 | 裁剪输入源固定为 `spriteFileId` |
| 4 | 对立绘结果继续裁头像 | 写入 `avatarFileId` 和 `avatarCropContext` |
| 5 | 绑定或创建立绘组 | 写入 `variantId`，必要时套用组配置 |

## 交互要求

1. 裁剪交互是明确的两阶段流程：立绘校正、头像校正。
2. 进入头像校正阶段时，如果当前头像没有 `spriteFileId`，提示“当前没有可用的立绘进行头像裁剪，请先完成立绘裁剪。”
3. 批量上传和单个上传都必须进入同一套裁剪流程。
4. 选择立绘组后，编辑行为等价于对组内头像进行批量应用。

## 桌面端与 WebGAL 要求

1. WebGAL 普通立绘优先使用 `spriteFileId` 派生资源。
2. 立绘组合成时使用基准头像的 `spriteFileId` 与当前头像的 `avatarFileId`。
3. 小头像资源由 `avatarFileId` 派生。
4. 发布包内的资源路径使用打包后的相对路径，例如 `avatarPath`，不把角色实体字段命名为图片地址。

## 验收标准

1. 新上传角色图片后，头像记录至少能看到 `originFileId`。
2. 完成立绘校正后，头像记录能看到 `spriteFileId` 和 `spriteCropContext`。
3. 完成头像校正后，头像记录能看到 `avatarFileId` 和 `avatarCropContext`。
4. 头像裁剪源来自立绘裁剪结果，而不是上传原图。
5. 角色列表、头像列表、WebGAL 发布均通过 fileId 派生展示资源。
