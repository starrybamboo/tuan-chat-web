# 文档房间 Message Stream 设计

## 背景

新的文档模块不再使用独立的文档协作模型，也不再以前端 BlockSuite 文档作为主数据结构。文档和聊天室在底层统一为同一种 `room + message stream` 模型：

- 聊天室是一个 room，以聊天视图展示消息流。
- 文档也是一个 room，以文档视图展示消息流。
- 二者的差异主要在 view 和同步策略，而不是数据结构。

这让同一批消息可以在聊天室视图和文档视图之间切换。所谓“切换到文档视图”不是把房间永久转换为另一种实体，而是用另一种 UI/UX 解释同一个 room 的 message stream。

## 设计目标

1. 文档和聊天室共享底层消息模型。
2. 文档和聊天室共享本地消息缓存。
3. 文档不做多人实时共同编辑。
4. 文档同步更 lazy，以批量 patch 为主。
5. 聊天室仍保持即时发送和即时同步。
6. 文档视图不能因为切换 view 清空或覆盖聊天室消息。
7. 本地未确认消息需要被统一识别，不能靠“文档逻辑猜聊天室逻辑”。

## 已确认的设计决策

这些是讨论过程中已经明确下来的约束，后续实现不应反复偏离：

1. 前端去掉 BlockSuite 作为文档主编辑模型。后端旧表和旧字段可以暂时保留，但新前端链路不再依赖 BlockSuite。
2. 不新增独立的 `doc_room` 表。文档就是一种 room，通过 `room.room_type` 表达默认展示倾向。
3. 不再使用 `sdoc:123:description` 这类 description 占位引用。空间/房间描述回到纯文本 `description`。
4. `message.content` 不升级为 `TEXT`。消息长度限制是产品和渲染边界的一部分，WebGAL 本身不能渲染过长消息。
5. 文档 view 不使用 `/chat/{spaceId}/doc/{roomId}` 这种独立路由语义，统一回到 `/chat/{spaceId}/{roomId}`，通过 room 信息和当前 view mode 决定渲染方式。
6. 文档同步不走独立 `/doc-room/{roomId}/sync` 主链路，归一到 room/message 的 history 与 patch 接口。
7. 文档 patch 接口是 `POST /chat/message/patch`，请求体携带 `roomId`，新增、更新、移动、删除都合并到 patch 语义里。
8. 文档本地存储使用项目统一的 room message cache，不另起一套文档 snapshot 主缓存。
9. 文档不是多人实时共同编辑场景，不引入 CRDT 或逐字实时同步。
10. 文档和聊天室的区别只在 view 和 workload：聊天室即时提交，文档 lazy 批量提交。

## 核心模型

底层统一抽象为：

- `room`：消息流容器。
- `room.type`：标识默认展示倾向，例如普通房间或文档房间。
- `message stream`：room 内按 position 排序的一组消息。
- `view mode`：当前 UI 选择用聊天方式还是文档方式渲染消息流。

`room.type` 决定默认打开时的体验；`view mode` 决定当前临时展示方式。用户从聊天室切到文档视图，本质是切换 view mode，不应创建新文档实体，也不应复制一份消息。

`room.type` 和 `view mode` 的关系：

- `room.type` 是持久化属性，用来表达这个 room 默认更像聊天室还是文档。
- `view mode` 是当前前端状态，用来表达用户这一刻想用聊天 UI 还是文档 UI 看同一批消息。
- 用户在普通聊天室里临时切到文档视图，不应该修改 `room.type`。
- 用户创建真正的文档房间时，可以让 `room.type` 默认为文档类型，但底层仍然是 message stream。

## 聊天室与文档的区别

聊天室：

- 面向多人实时交流。
- 用户发送一条消息，就立即进入本地缓存。
- 前端进行乐观更新，让 UI 先显示。
- 服务端返回后，用真实消息替换本地临时消息。
- 其他用户的新消息会通过现有消息同步链路进入同一份缓存。

文档：

- 面向单人编辑。
- 不需要多人实时协同。
- 编辑动作不会每次都立即提交到云端。
- 前端先修改同一份本地 message stream。
- 等到空闲、延迟、离开页面等时机，再把改动批量 patch 到云端。

因此二者不是“数据不同”，而是“写入节奏不同”：

- 聊天室：即时新增。
- 文档：lazy 批量 patch。

## 本地缓存原则

文档视图和聊天室视图必须共用同一套 room message cache。

这意味着：

1. 聊天视图看到的消息，切到文档视图后应该仍然存在。
2. 文档视图编辑后的消息，切回聊天视图后也应该反映在同一份消息流中。
3. 不能为文档单独维护一套和聊天室不互通的 `messageDraft` 或 snapshot 主缓存。
4. 本地 room message cache 是云端 room message stream 的本地镜像；Web 端缓存可用性取决于 sqlite-wasm OPFS 初始化结果。

文档可以保留少量编辑器运行时状态，例如光标、选区、块的临时 UI ID，但这些不能成为独立的数据源。

文档编辑器内部可以有临时块模型，但它只是 UI 投影：

- 从 room message cache 读出消息。
- 转成编辑器块用于渲染和编辑。
- 编辑后再转回同一套 message stream。
- 最终写入本地缓存和云端 patch 的仍然是 message，而不是 `messageDraft`、BlockSuite doc 或独立 snapshot。

## 乐观消息状态

聊天室发送消息时会先创建一条本地临时消息，这叫乐观消息。

它的作用是：

1. 用户发送后 UI 立刻显示。
2. 网络请求仍在等待服务端确认。
3. 服务端返回真实消息后，本地临时消息被真实消息替换。

为了让聊天室和文档统一识别这种状态，消息对象上会带一个明确的本地状态：

```text
tcLocalSyncState = optimistic
```

它表示：

- 这条消息已经存在于本地 UI 和缓存。
- 但它还没有被服务端确认为真实消息。
- 它不能被文档 patch 当成新的文档块再次提交。

负数 message id 仍然可能存在，但它只是本地临时主键，用来在服务端返回后找到并替换对应消息。业务语义不再依赖“负 ID 就是乐观消息”，而是优先看显式本地状态。

为了兼容旧缓存，系统仍会把负 ID 当作乐观态兜底识别，但新的统一语义应该是本地状态字段。

## 文档 patch 同步

文档视图编辑消息流后，不做全量覆盖，而是基于基线消息流和当前消息流生成 patch。

基线消息流：

- 进入文档视图时读取到的当前消息列表。
- 或上一次成功同步后服务端返回的消息列表。

当前消息流：

- 用户在文档视图中编辑后的本地消息列表。

patch 操作类型：

- `insert`：当前有、基线没有的消息。
- `update`：同一 message id 的内容或元数据变化。
- `move`：同一 message id 的 position 变化。
- `delete`：基线有、当前没有的消息。

文档 patch 的特殊规则：

- 本地乐观消息不参与 patch。
- 新建文档块没有真实 message id 时，会作为 insert。
- 已持久化消息通过真实 message id 做 update、move、delete。
- 同步成功后，用服务端返回的消息刷新基线。

这样可以避免输入一行就全量覆盖整个 room，也能避免把聊天室未确认消息重复插入云端。

patch 是当前唯一的文档云端写入主语义。它同时覆盖原先可能拆开的批量新增和编辑同步：

- `insert` 用于新文档块或文档视图批量新增的消息。
- `update` 用于内容、角色、头像、消息类型、额外元数据等变化。
- `move` 用于文档块重排。
- `delete` 用于删除消息块。

服务端负责在一个事务里应用 patch，并返回应用后的权威消息流或必要的消息结果。前端成功后更新本地 room message cache 和同步基线。

### 为什么不用全量 sync

全量 sync 的问题是破坏同构模型：

1. 输入一行就提交整个 room，频率和数据量都不合理。
2. 如果加载过程拿到空 snapshot，容易把原有聊天室消息覆盖没。
3. 当聊天室乐观消息和文档编辑并存时，全量覆盖很难表达“只提交文档改动，不提交未确认消息”。
4. 全量写入会让文档视图像一个独立实体，而不是同一份 message stream 的另一种 view。

因此文档只提交差异 patch，本地缓存仍然先行更新。

## 同步时机

聊天室同步时机：

- 用户发送后立即提交。
- WebSocket 或历史接口持续补齐远端消息。
- 乐观消息在服务端确认后立即回填。

文档同步时机：

- 用户编辑后先写本地状态。
- 延迟一段时间后批量 patch。
- 离开或卸载编辑器时尝试 flush。
- 没有实际变化时不提交。
- 空文档保护：避免因为加载或切换异常把已有内容清空。

文档同步应该偏 lazy，目标是减少请求频率，同时保证离开页面前尽量落云。

lazy 的具体含义：

- 单次输入不应立即触发一次远端请求。
- 连续编辑应该被 debounce 或批处理合并。
- 切换 view、切换 room、页面卸载前可以主动 flush。
- 如果 patch 队列为空，不能发空请求。
- 如果本地只有显式乐观态消息，不应把它们当作文档新增提交。

## 视图切换

从聊天室切到文档视图：

1. 不改变 room 的永久类型。
2. 不创建新的文档路由。
3. 从同一份 room message cache 读取消息。
4. 用文档编辑器方式渲染消息流。
5. 编辑时只改变同一份消息流。

从文档视图切回聊天室：

1. 不重新拉一份独立文档。
2. 仍然使用同一份 room message cache。
3. 聊天 UI 按消息顺序展示当前消息流。

所以“转换成文档视图”只是当前 room 的临时 view 切换，而不是数据迁移或实体转换。

视图切换时的用户预期：

- 聊天室里已有的消息，切到文档视图必须仍然可见。
- 文档视图编辑后切回聊天室，聊天室必须看到同一批消息的最新内容。
- 切换过程中可以有 UI 过渡，但不应该因为重新加载编辑器导致明显空白或内容闪没。
- 切换 view 不应该触发任何删除、清空、全量覆盖。

## 冲突模型

当前文档不支持多人实时共同编辑，因此冲突模型可以保持简单：

- 文档编辑默认只由当前用户修改。
- 同一个 room 如果同时作为聊天室接收其他消息，新消息仍进入共享缓存。
- 文档 patch 只处理当前用户在文档视图产生的差异。
- 本地乐观消息先跳过，等服务端确认后再进入可编辑/可 patch 的持久化消息集合。

未来如果需要多人文档协作，不能直接在当前 patch 模型上模拟实时协作，应重新引入明确的并发编辑协议。

## 数据安全保护

为避免“切到文档视图后内容消失”，需要遵守这些保护：

1. 文档视图不得以空 snapshot 覆盖 room cache。
2. 远端返回空消息时，不应直接清空本地 room cache。
3. patch 成功后以服务端返回消息作为新基线。
4. 本地乐观消息不当作文档新增块提交。
5. 文档视图加载失败时保留已有本地消息。
6. view 切换不做删除或清空动作。

## BlockSuite 迁移策略

旧 BlockSuite 文档可以通过一次性脚本迁移，不需要在服务端运行时实时转换。

迁移思路：

1. 扫描旧 BlockSuite 文档数据。
2. 提取可表达为消息的内容块。
3. 转换为 room message stream。
4. 写入对应 room。
5. 更新 room 的文档相关元数据。
6. 迁移完成后，前端只读取新的 message stream。

迁移脚本应尽量幂等，避免重复写入同一批消息。

迁移不是运行时兼容层。迁移完成后的前端只认 room message stream：

- 旧 BlockSuite 数据作为输入源。
- 脚本把可表达内容转换成一组 message。
- 写入对应 room 后，后续编辑只走 message stream patch。
- 如果旧文档无法无损表达为消息，应保留可读文本优先，不阻塞新模型落地。

## 非目标

当前阶段明确不做：

1. 不做多人文档实时协作。
2. 不把文档重新拆成独立文档实体和独立缓存。
3. 不为文档维护独立 `/doc-room` 同步协议作为主链路。
4. 不通过全量 snapshot 覆盖 room message stream。
5. 不放宽单条消息长度限制来承载整篇长文。
6. 不把 `description` 重新变成 BlockSuite 或 sdoc 引用。
7. 不用负数 message id 作为唯一乐观态判断；负 ID 只是兼容兜底。

## 当前完成状态

当前设计已经落到核心链路：

- 文档和聊天室底层统一为 room message stream。
- 文档视图通过同一份 room message cache 渲染。
- 文档同步使用 `POST /chat/message/patch`。
- 聊天室乐观消息和文档 patch 通过统一本地状态协调。
- 文档 view 切换不再代表永久转换。

后续重点不是再拆新模型，而是继续把边缘入口收敛到这套统一语义上，例如导入、批量新增、WebGAL 生成、旧文档迁移等流程，都应该最终写入同一套 room message stream。
