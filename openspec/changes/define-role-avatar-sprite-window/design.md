## Context

角色详情页的头像区域会打开一个全屏/大弹窗，用于管理同一角色下的头像、立绘、立绘组和裁剪上下文。该窗口的复杂度来自三件事：

- “头像”不是独立上传结果，而是从“立绘裁剪结果”继续裁剪出来的小图。
- “立绘组”不是普通筛选标签，而是同一套 WebGAL 合成配置和裁剪上下文的批量对象。
- 上传、批量导入、拖拽入组、绑定已有组、新建组、回收站恢复都会改变头像列表和组关系。

当前实现中的主要组件边界：

- `SpriteSettingsPopup`：窗口 shell、tab 状态、头像/立绘组选择状态、批量对象编排。
- `SpriteListGrid`：左侧头像列表、分类折叠、上传 tile、多选、拖拽、删除。
- `AvatarSettingsTab`：单头像名称、分类、立绘组绑定、WebGAL/聊天预览。
- `SpriteCropper`：立绘校正和头像校正的裁剪 UI、单体/批量应用、立绘组配置更新。
- `AvatarUploadCropper`：上传入口里选择未分组、已有立绘组或新建立绘组。
- `avatarCropContext`：裁剪上下文与立绘组合成配置之间的转换。

现有产品共识：

- 一个头像只能属于一个立绘组。
- 立绘组与 category 相互独立；组内查看仍按 category 分类。
- 裁剪只裁原图；压缩是裁剪结果的媒体派生，不是裁剪输入。
- 头像裁剪必须从立绘裁剪结果开始，不能直接从上传原图裁头像。
- 立绘校正完成后必须继续头像校正。

## Goals / Non-Goals

**Goals:**

- 固化这个窗口内角色、头像、立绘、立绘组的交互和数据语义。
- 明确单选、多选、立绘组选择、组内管理四种选择态的区别。
- 明确上传导入和裁剪流程统一为“原图上传 -> 立绘校正 -> 头像校正”。
- 明确立绘组的编辑等价于对组内头像批量应用，并同步维护 WebGAL transform。
- 明确失败、部分成功、删除恢复和进度反馈的最低要求。

**Non-Goals:**

- 不设计新的后端 API；只记录前端窗口对现有 API 和模型的要求。
- 不恢复旧 URL 字段兼容逻辑。
- 不实现多层立绘差分编辑；当前只规范 origin -> sprite -> avatar 两阶段裁剪。
- 不定义移动端角色编辑的完整页面，只要求同一窗口在移动端保持等价流程。

## Decisions

### 1. 立绘组选择态与组内视图分离

点击左侧立绘组卡片只选中该组，并把右侧切到立绘组批量设置。只有点击“进入组内管理”后，左侧列表才切到该组目录。

理由：立绘组在外层承担“文件夹/批量对象”角色，点击它应该像点击普通头像一样选中对象，而不是直接改变浏览位置。显式进入组内可以避免用户找不到返回路径。

替代方案：点击组直接进入组内。已被否定，因为用户难以回到外层，且不符合“点击对象后右侧编辑”的窗口模式。

### 2. 头像列表的 category 与 variant 独立

category 是头像的分类展示维度；variant 是立绘组归属。外层展示未入组头像和立绘组卡片；进入某个立绘组后，组内头像仍按 category 折叠。

理由：category 用于用户整理命名，variant 用于 WebGAL 合成配置。混在同一层会导致“文件夹”和“分类”语义冲突。

### 3. 裁剪状态机强制两阶段

上传或选择原图后必须先完成立绘校正，写入 `spriteFileId` 和 `spriteCropContext`；之后必须进入头像校正，从 `spriteFileId` 继续裁剪并写入 `avatarFileId` 和 `avatarCropContext`。

理由：头像是从立绘裁剪结果来的。如果允许从原图直接裁头像，立绘组配置和 WebGAL 合成槽位会失去一致来源。

### 4. 立绘组配置是批量编辑的单一真相

立绘组的 `compositionConfig` 存储同组共享的 `spriteCrop`、`avatarSlot`、`spriteTransform` 和输出画布信息。编辑一个立绘组时，裁剪器使用组选中头像集合，并在应用时更新组配置和组内头像结果。

理由：立绘组用于批量合成；如果只改头像结果而不改组配置，WebGAL 预览、后续绑定、组内新上传都会不一致。

### 5. 绑定已有立绘组会放弃单头像旧上下文

头像绑定到已有立绘组时，系统使用目标组配置重建立绘和头像结果，并用目标组的 `spriteTransform` 覆盖单头像旧 transform。

理由：归类到某个组意味着服从该组统一合成上下文。保留旧上下文会让同组输出不可预测。

### 6. 上传入口统一目标选择

单个上传和批量上传都必须支持：未分组、新建立绘组、绑定已有立绘组。无论入口来自外层、category、还是组内，都进入同一套裁剪流程。

理由：用户不应因为入口不同而得到不同流程；否则会出现部分数据缺少 `spriteCropContext` 或 `avatarCropContext`。

### 7. 部分成功必须可见

批量上传、批量裁剪、批量绑定可以部分成功，但 UI 必须显示成功/失败数量，并只让成功项继续下一步。

理由：媒体上传和裁剪是长链路，不能用一次 500 或一个 toast 让用户失去成功项和失败项的判断。

## Risks / Trade-offs

- 立绘校正后强制进入头像校正会增加用户操作步数 → 通过清晰提示和保留当前选择集降低成本。
- 批量绑定已有立绘组需要加载原图、裁立绘、再裁头像，耗时较长 → 必须显示进度和部分失败统计。
- 立绘组要求原图尺寸一致，会阻止部分旧数据直接入组 → 需要给出“尺寸不一致/缺少原图/缺少立绘”的具体错误。
- 组配置是单一真相，误操作会批量影响组内头像 → 立绘组设置页必须明确显示组内数量，并提供进入组内管理入口供用户确认。
- 外层选中组和进入组内视图是两个状态 → 状态命名和 UI 文案必须清晰，否则容易再次混淆。

## Migration Plan

1. 将本 change 作为审查稿，不直接改变生产行为。
2. 审查通过后，把 `role-avatar-sprite-window` capability 同步到正式 specs。
3. 对照 tasks 检查现有实现，补齐不满足的交互与测试。
4. 旧 URL fallback、旧 variant 数据兼容不纳入迁移；旧数据只通过显式校正/重建进入新流程。

## Open Questions

- 立绘组外层是否应该显示“未分组”虚拟文件夹，还是仅显示真实立绘组 + 未分组头像列表？
- 批量立绘校正后弹“是否创建立绘组”时，是否应该阻塞头像校正，还是允许背景已进入头像校正？
- 绑定已有立绘组时，如果个别头像原图尺寸不一致，是否允许成功项先入组，失败项保留未分组？
- 删除立绘组时，组内头像应默认移回未分组，还是要求用户先清空组？
